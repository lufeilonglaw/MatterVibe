# MatterVibe AI 接口说明

> 把本文档直接喂给你的 AI 助手（Marvis、小龙虾 OpenClaw／QClaw／StepClaw 等），
> 它就学会了读写 MatterVibe 案件看板。也可将本文件放入 OpenClaw 的技能目录作为 SKILL。

## MatterVibe 是什么

**MatterVibe 是一款面向中国执业律师的本地案件管理（看板）软件**，由律师本人主导、以 AI 协作方式
开发，开源、免费、数据全部存在本机（Electron + 单文件 SQLite，不上传任何服务器）。它的核心理念是
把每一个案件做成一张"案卷看板"：

- **案件看板**：每个案件有一列"案卷封皮"（案号、当事人、法院、承办法官、关键期限等要素），
  以及若干"办案阶段"列（如收案准备、立案管辖、证据庭前、开庭、裁判后续），每个阶段下是可勾销的
  任务卡片。律师像在真实卷宗上推进工作一样，逐条划销。
- **办案模板**：内置民商事、刑事、行政、执行、非诉五套阶段模板，体现常见办案流程；可自定义、
  可导入/导出分享（"模板就是办案思路"）。
- **首页待办总览**：跨案件汇总今日/本周开庭、临近期限、待办事项。
- **日历**：月视图展示开庭、举证、调解、自定义事件，标注节假日。
- **办案日志与邮寄记录**：记录每次工作与文书寄送。
- **期限提醒**：扫描封皮里的日期，临期在顶栏铃铛与系统通知提醒。
- **律师工具箱**：诉讼费/保全费/风险代理费/仲裁费计算、法定期限届满日（节假日顺延）、金额大写等。
- **多设备同步**：可通过坚果云等在 Mac/Windows 间端到端加密同步。
- **本地 AI 接口**（就是你正在用的这个）：让 AI 助手帮律师把案件材料录入看板。

## 你的角色：能做什么

你可以通过本地 HTTP 接口：查看案件、新建案件、填写案件封皮信息、调整办案阶段、向阶段添加任务、
划销任务、记办案日志、记文书邮寄记录、添加日程事件、查询临近的法定期限。

典型用法：律师把一份判决书、起诉状或案情材料交给你分析后，你把分析结果落进 MatterVibe——
先就案件类型与阶段划分向律师确认（见下方"建案工作流"）→ 新建或定位案件 → 更新封皮要素
（案号、当事人、关键期限日）→ 把待办事项逐条写入对应阶段 → 把关键日期建为日程事件 →
记一条办案日志说明本次处理内容 → 向律师汇报录入了什么。

你的价值在于帮律师**高效地把非结构化的案件材料，转化为结构化的看板**，但**阶段如何划分、
任务如何归类，体现的是律师的执业判断**——你负责高效执行和提出建议，关键判断先问律师、由其拍板。

## 连接方式

- 地址：`http://127.0.0.1:2046/api`（端口以软件内「设置 → AI 接口」显示为准，默认 2046）
- **字符编码：请求体必须以 UTF-8 编码发送**，并带 `Content-Type: application/json; charset=utf-8`。
  含中文时尤其重要：用 Python 请用 `json.dumps(data, ensure_ascii=False).encode("utf-8")`；
  **不要用 PowerShell 的 `Invoke-RestMethod -Body "中文字符串"`**（它默认按系统 ANSI 编码发送，
  会导致中文乱码）——若必须用 PowerShell，请显式将 body 转为 UTF-8 字节再发送。
- 鉴权：每个请求必须携带请求头 `Authorization: Bearer <令牌>`
  （令牌在 MatterVibe 设置 → AI 接口中查看；接口默认关闭，需律师手动开启）
- 所有响应均为 JSON：`{ "ok": true, "data": ... }` 或 `{ "ok": false, "error": "原因" }`
- 接口处于只读模式时，全部 POST 返回 403——此时只查询、不写入，并告知用户。

## 接口清单

### 查询（GET）
| 接口 | 说明 |
|---|---|
| `/api/health` | 探活，返回软件版本 |
| `/api/matters` | 案件列表（id、name、type、icon、remind） |
| `/api/matters/{id}` | 案件详情：封皮 cover_info（[要素名, 值] 数组）、阶段 stages 及其中任务 tasks |
| `/api/templates` | 可用的建案模板（key、name、type、description） |
| `/api/deadlines` | 已开启提醒的案件中扫描到的全部日期（matter_id、matter_name、label、date） |
| `/api/agenda?from=YYYY-MM-DD&to=YYYY-MM-DD` | 指定区间的全部日程（结构化事件＋封皮日期合并） |
| `/api/events?from=&to=` | 指定区间的结构化事件 |
| `/api/logs/{matterId}` | 某案件的办案日志 |
| `/api/mails/{matterId}` | 某案件的邮寄记录 |

### 写入（POST，JSON 请求体）
| 接口 | 请求体 | 说明 |
|---|---|---|
| `/api/matters` | `{"name":"案件名","template":"zhixing"}` | 新建案件。模板 key：zhixing 执行／minshangshi 民商事／xingshi 刑事／feisu 非诉／xingzheng 行政 |
| `/api/matters/{id}/cover` | `{"updates":[["案号","（2026）×0112执123号"],["上诉截止日","2026-06-28"]]}` | 更新封皮：要素名存在则更新值，不存在则追加新要素 |
| `/api/stages` | `{"matter_id":1,"name":"📂 二审阶段"}` | 添加阶段列 |
| `/api/tasks` | `{"matter_id":1,"stage_name":"财产","content":"申请调查令"}` 或 `{"stage_id":5,"content":"..."}` | 添加任务。stage_name 为健壮匹配：自动忽略阶段名前的图标（如 📂）与空格、双向包含，传纯文字即可命中；若匹配不上，接口会在错误信息里**返回该案件所有可用阶段名**，照着改即可 |
| `/api/tasks/{id}/complete` | `{"done":true}` | 划销／恢复任务 |
| `/api/logs` | `{"matter_id":1,"content":"AI 拆解判决书生成任务清单","hours":0.2,"date":"2026-06-13"}` | 记办案日志（hours、date 可省略） |
| `/api/mails` | `{"matter_id":1,"recipient":"××区人民法院执行局","contents":"调查令申请书","courier":"EMS","tracking_no":"EA×××CN"}` | 记邮寄记录 |
| `/api/events` | `{"matter_id":1,"event_date":"2026-06-28","event_time":"09:30","kind":"hearing","title":"第一次开庭"}` | 添加日程事件（会出现在首页与日历）。kind：hearing 开庭／evidence 举证提交／mediation 调解续封／custom 自定义 |

## curl 示例

> 以下 curl 示例在带中文时，请确保终端与文件为 UTF-8 环境。推荐用 Python（见文末）。

```bash
TOKEN=把设置里的令牌粘在这里
# 列出案件
curl -s http://127.0.0.1:2046/api/matters -H "Authorization: Bearer $TOKEN"
# 给某执行案的财产阶段加一条任务
curl -s -X POST http://127.0.0.1:2046/api/tasks \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"matter_id":3,"stage_name":"财产","content":"申请调查令——被执行人支付宝流水"}'
```

## 案件命名规范（新建案件时遵循）

新建案件时，`name` 字段按案件类型采用统一命名风格，便于律师在看板和首页快速辨识：

- **民商事 / 行政等诉讼案件**：用「当事人 + 案由」。当事人名称较长时可用简称或缩写。
  - 例：`西姆 VS 东方惠尔·商业诋毁纠纷`、`张某诉李某·民间借贷纠纷`
- **刑事案件**：用「当事人 + 涉嫌 + 罪名」。
  - 例：`王某涉嫌职务侵占`、`赵某涉嫌合同诈骗`
- **非诉 / 顾问类**：用「客户 + 事项」。
  - 例：`某科技公司·常年法律顾问`、`某集团·股权收购尽调`

命名保持简洁，避免把完整公司全称、案号都堆进标题（这些放进封皮要素即可）。

## 阶段划分指导（建案后按案件实际情况组织阶段）

出厂模板提供的是**通用程序骨架**，不预设原告/被告等具体立场。真实案件的阶段划分因
诉讼地位、审级、程序类型而异，因此建案后你可以用 `POST /api/stages` 调整阶段结构
（新增缺失的阶段），再把任务填进对应阶段。下面是常见案件类型的推荐阶段结构，供参考：

**民事/商事 一审**
- 通用骨架（原告、被告通用）：收案与准备 → 立案与管辖 → 证据与庭前 → 开庭审理 → 裁判与后续
- 原告方可强调：诉前保全、起草起诉状、立案缴费
- 被告方可强调：研究起诉状与证据、起草答辩状、（如需）管辖权异议、反诉评估

**民事/商事 二审**
- 收案与上诉/答辩 → 二审证据与争点整理 → 二审开庭或书面审 → 裁判与再审评估

**刑事辩护**（按程序阶段，立场中立）
- 侦查阶段（黄金37天）→ 审查起诉阶段 → 审判阶段 → 判决与归档
- 注意当事人当前所处阶段，只在对应阶段及之后填任务

**行政诉讼/复议**
- 诉前复议与期限 → 立案与举证（被告负举证责任）→ 程序与依据审查 → 庭审 → 裁判与执行监督

**仲裁**
- 收案与仲裁申请/答辩 → 仲裁员选定与管辖 → 证据交换与开庭 → 裁决与撤裁/执行

**执行**
- 收案与手续 → 财产调查与线索 → 法院沟通与材料 → 博弈/投诉/衍生诉讼 → 结案

以上为参考，不必机械照搬。核心原则：**阶段反映该案真实的程序推进路径**，任务落在
它真实应处的阶段里，不要硬塞进不匹配的模板阶段。

## 建案工作流（重要：先确认，再录入）

为避免把阶段结构搞乱，录入一个新案件时请按以下顺序，**关键判断先向用户提问确认，
不要自行假设**：

1. **读取并分析案件材料**，提取：案件类型、当事人、我方诉讼地位（原告/被告/申请人/
   被申请人/辩护）、审级、案号、法院、关键日期、争议焦点等。
2. **就阶段划分向用户提问确认**，例如：
   - "这个案子我方是原告还是被告？"
   - "是一审还是二审？需要单独的『管辖权异议』阶段吗？"
   - "刑事案件目前处于侦查、审查起诉还是审判阶段？"
   - "需要保全/财产调查阶段吗？"
   把你打算采用的阶段结构（如"收案与准备 → 立案与管辖 → …"）列给用户看，请其确认或修改。
3. **用户确认后**：`POST /api/matters` 建案 → 按确认的结构用 `POST /api/stages` 补齐
   缺失阶段 → `POST /api/tasks` 把任务填进对应阶段（用 stage_name 即可，匹配会忽略图标）。
4. **填写封皮**（`/api/matters/{id}/cover`）、**关键日期建为事件**（`/api/events`）、
   必要时**记录办案日志**（`/api/logs`）。
5. 录入完成后，向用户**汇报本次录入了哪些内容**（阶段数、任务数、关键日期），便于其核对。

> 当你不确定阶段该怎么分、某条任务该归到哪个阶段时，**宁可先问用户，不要乱建**。
> 阶段划分体现的是律师的执业判断，你的角色是高效执行 + 提出建议，最终结构由用户拍板。

## 行为准则（重要）

1. **写入前先读**：先 GET 案件详情确认阶段结构，再决定任务落在哪个阶段，避免重复添加。
2. **每次批量写入后记一条办案日志**，说明你做了什么（如"AI 根据 X 判决书生成 8 条执行任务"），让律师可追溯。
3. 拆解判决书／起诉书时，把开庭日、举证截止日等**关键日期用 `/api/events` 录成结构化事件**（带 kind 与时间），它们会自动出现在律师的首页待办与日历中。
4. **不确定时只读不写**，把建议列给律师确认。
4. 涉及当事人隐私的内容，不要复述到 MatterVibe 之外的任何输出渠道。
